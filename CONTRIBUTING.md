# Contributing to rules_ocx

Thanks for helping out! Ground rules:

## Workflow

- Conventional Commits (`feat:`, `fix:`, `docs:`, `test:`, `ci:`, `chore:`).
  The changelog is generated by git-cliff from commit messages.
- Branch from `main`, open a PR. Never push to `main` directly.
- `task verify` must pass locally before requesting review. It runs the
  linters plus `bazel test //...` and the example workspaces.

## Design invariants

- **Never re-implement OCX internals in Starlark.** No OCI protocol, no
  registry auth, no index/store layout knowledge. Everything goes through the
  pinned `ocx` CLI; if the CLI is missing a capability, file an issue against
  [ocx-sh/ocx](https://github.com/ocx-sh/ocx) instead.
- The pinned ocx version (`ocx/private/versions.bzl`) and the vendored
  manifest (`dist/dist.json`) are bumped together, deliberately.
- Host detection lives in repository rules only — module extension
  implementations must stay reproducible pure functions of their tags.

## Toolchain

The dev toolchain is provisioned by OCX itself (dogfooding): `ocx pull` or
`direnv allow` gives you bazelisk, actionlint, git-cliff, and friends from
the committed `ocx.toml`/`ocx.lock`.
